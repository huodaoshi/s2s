# 分拣（Triage）

将项目 issue 跟踪器上的 issue 在小型分拣角色状态机中流转。

分拣期间在 issue 跟踪器发布的**每条**评论或 issue **必须**以如下声明开头：

```
> *This was generated by AI during triage.*
```

## 参考文档

- [AGENT-BRIEF.md](AGENT-BRIEF.md) — 如何撰写耐久的 Agent 简报
- [OUT-OF-SCOPE.md](OUT-OF-SCOPE.md) — `.out-of-scope/` 知识库如何运作

## 角色

两类**类别**角色：

- `bug` — 某处已损坏
- `enhancement` — 新功能或改进

五类**状态**角色：

- `needs-triage` — 维护者需评估
- `needs-info` — 等待报告者补充信息
- `ready-for-agent` — 规格完整，可供 AFK Agent 领取
- `ready-for-human` — 需人工实现
- `wontfix` — 不予处理

每个已分拣 issue 应恰好携带一个类别角色和一个状态角色。若状态角色冲突，先向维护者标出并询问，勿做其他操作。

以上为规范角色名——issue 跟踪器中的实际标签字符串可能不同。映射关系应已提供给你；若未配置，请运行 `/setup-matt-pocock-skills`。

状态流转：未打标签的 issue 通常先进入 `needs-triage`；由此可转至 `needs-info`、`ready-for-agent`、`ready-for-human` 或 `wontfix`。报告者回复后，`needs-info` 回到 `needs-triage`。维护者可随时覆盖——对看起来异常的流转先标出并询问再继续。

## 调用方式

维护者调用 `/triage` 并用自然语言说明意图。理解请求后执行。例如：

- 「给我看需要我处理的事项」
- 「看看 #42」
- 「把 #42 移到 ready-for-agent」
- 「有哪些可以给 Agent 领取？」

## 展示待关注项

查询 issue 跟踪器，分三桶展示，每桶内按最旧优先：

1. **未打标签** — 从未分拣。
2. **`needs-triage`** — 评估进行中。
3. **`needs-info` 且自上次分拣备注以来报告者有新活动** — 需重新评估。

显示数量与每条一行摘要。由维护者选择处理哪条。

## 分拣指定 issue

1. **收集上下文。** 阅读完整 issue（正文、评论、标签、报告者、日期）。解析既有分拣备注，避免重复已解决的问题。用项目领域术语表探索代码库，尊重所涉 ADR。阅读 `.out-of-scope/*.md`，若与当前 issue 相近的既往驳回，一并告知维护者。

2. **给出建议。** 向维护者说明推荐的类别与状态及理由，并附与 issue 相关的简要代码库摘要。等待指示。

3. **复现（仅 bug）。** 拷问前先尝试复现：读报告者步骤、追踪相关代码、运行测试或命令。报告结果——成功复现及代码路径、未能复现、或信息不足（强 `needs-info` 信号）。已确认复现可写出更有力的 Agent 简报。

4. **拷问（若需要）。** issue 需充实细节时，运行 `/grill-with-docs` 会话。

5. **落实结果：**
   - `ready-for-agent` — 发布 Agent 简报评论（见 [AGENT-BRIEF.md](AGENT-BRIEF.md)）。
   - `ready-for-human` — 结构与 Agent 简报相同，但说明为何不能委派（判断、外部访问、设计决策、手工测试等）。
   - `needs-info` — 发布分拣备注（见下方模板）。
   - `wontfix`（bug）— 礼貌说明后关闭。
   - `wontfix`（enhancement）— 写入 `.out-of-scope/`，在评论中链接，然后关闭（见 [OUT-OF-SCOPE.md](OUT-OF-SCOPE.md)）。
   - `needs-triage` — 应用该角色；若有部分进展可附可选评论。

## 快速状态覆盖

若维护者说「把 #42 移到 ready-for-agent」，信任其判断并直接应用角色。先确认即将执行的操作（角色变更、评论、关闭），再执行。跳过拷问。若在未经过拷问的情况下移到 `ready-for-agent`，询问是否要撰写 Agent 简报。

## needs-info 模板

```markdown
## 分拣备注

**目前已确定：**

- 要点 1
- 要点 2

**仍需要您（@reporter）提供：**

- 问题 1
- 问题 2
```

拷问中已厘清的内容写入「目前已确定」，避免工作丢失。问题须具体、可执行，勿写「请提供更多信息」之类空泛要求。

## 续接先前会话

若 issue 上已有分拣备注，先阅读，检查报告者是否已回答待决问题，再呈现更新后的情况并继续。勿重复已解决的问题。
