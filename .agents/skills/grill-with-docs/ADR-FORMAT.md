# ADR 格式

ADR 放在 `docs/adr/`，按序号命名：`0001-slug.md`、`0002-slug.md` 等。

`docs/adr/` 目录按需创建——仅在需要首条 ADR 时建立。

## 模板

```md
# {决策的简短标题}

{1–3 句：背景是什么、决定了什么、为何如此。}
```

仅此而已。一条 ADR 可以只有一段。价值在于记录**做出了什么决定**以及**为何**——不在于填满章节。

## 可选章节

仅在有实质价值时加入；多数 ADR 不需要。

- **Status** frontmatter（`proposed | accepted | deprecated | superseded by ADR-NNNN`）——决策可能被重新审视时有用
- **Considered Options**——仅当被拒替代方案值得记住
- **Consequences**——仅当非显而易见的下游影响需要点明

## 编号

扫描 `docs/adr/` 取现有最大编号，加一。

## 何时提议 ADR

须**同时**满足以下三条：

1. **难以逆转**——日后改主意成本显著
2. **脱离上下文会令人意外**——未来读者看代码会纳闷「为何这样搞？」
3. **源于真实权衡**——确有可行替代方案，且因具体理由选了当前方案

若决策易于推翻，不必记——直接改即可。若不令人意外，无人会追问缘由。若无真实替代，除「做了显而易见的事」外无需记录。

### 适合记录的情形

- **架构形态。** 「我们用 monorepo。」「写模型 event-sourced，读模型投影到 Postgres。」
- **上下文之间的集成模式。** 「Ordering 与 Billing 通过领域事件通信，而非同步 HTTP。」
- **带来锁定的技术选型。** 数据库、消息总线、认证提供方、部署目标等——不必每条库都记，只记换掉要耗一季度的那些。
- **边界与范围决策。** 「客户数据归 Customer 上下文所有；其他上下文仅按 ID 引用。」明确的「不做」与「要做」同样重要。
- **有意偏离显而易见路径。** 「因 X 用手写 SQL 而非 ORM。」任何合理读者会假设相反做法的，都应记下，避免下一位工程师把刻意设计「修」掉。
- **代码中不可见的约束。** 「因合规不能用 AWS。」「因合作方 API 合约，响应须低于 200ms。」
- **拒绝理由非显而易见的备选方案。** 若曾考虑 GraphQL 却因微妙原因选 REST，应记录——否则半年后又会有人提议 GraphQL。
