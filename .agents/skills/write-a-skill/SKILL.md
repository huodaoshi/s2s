# 撰写技能

## 流程

1. **收集需求** — 向用户询问：
   - 技能覆盖什么任务/领域？
   - 应处理哪些具体用例？
   - 需要可执行脚本，还是仅指令？
   - 是否包含参考材料？

2. **起草技能** — 创建：
   - 简明指令的 `SKILL.md`
   - 内容超过约 500 行时另建参考文件
   - 需确定性操作时加工具脚本

3. **与用户评审** — 展示草稿并问：
   - 是否覆盖你的用例？
   - 有无遗漏或不清晰？
   - 哪些章节应更详/更简？

## 技能结构

```
skill-name/
├── SKILL.md           # 主指令（必需）
├── REFERENCE.md       # 详细文档（可选）
├── EXAMPLES.md        # 用法示例（可选）
└── scripts/           # 工具脚本（可选）
    └── helper.js
```

## SKILL.md 模板

```md
---
name: skill-name
description: 能力简述。在 [具体触发条件] 时使用。
---

# 技能名称

## 快速开始

[最小可用示例]

## 工作流

[复杂任务的步骤与检查清单]

## 高级功能

[链接到独立文件：见 [REFERENCE.md](REFERENCE.md)]
```

## description 要求

`description` 是 Agent **决定是否加载该技能时唯一可见的字段**。它与所有已安装技能一并出现在系统提示中。Agent 读这些描述，根据用户请求选技能。

**目标**：让 Agent 刚好知道：

1. 本技能提供什么能力
2. 何时/为何触发（具体关键词、上下文、文件类型）

**格式**：

- 最多 1024 字符
- 第三人称撰写
- 第一句：做什么
- 第二句：「在 [具体触发条件] 时使用」

**好例子**：

```
从 PDF 提取文本与表格、填表、合并文档。在处理 PDF 或用户提到 PDF、表单、文档提取时使用。
```

**差例子**：

```
帮助处理文档。
```

差例子无法与其它文档技能区分。

## 何时加脚本

在以下情况加工具脚本：

- 操作确定性（校验、格式化）
- 同一段代码会反复生成
- 错误需显式处理

相对现场生成代码，脚本省 token、更可靠。

## 何时拆文件

在以下情况拆成独立文件：

- `SKILL.md` 超过 100 行
- 内容属不同领域（如财务 schema vs 销售 schema）
- 高级功能很少用到

## 评审检查清单

起草后核对：

- [ ] description 含触发条件（「在…时使用」）
- [ ] `SKILL.md` 少于 100 行
- [ ] 无时效性信息
- [ ] 术语一致
- [ ] 含具体示例
- [ ] 引用仅一层深（勿链式套娃）
