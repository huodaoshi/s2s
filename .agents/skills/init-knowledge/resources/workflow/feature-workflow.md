# 功能工作流（feature-workflow）

Agent 做功能相关工作时，**先判断工作类型**（见下文），**只跟对应章节**，不要混用多条流程。

Issue 约定见 `docs/agents/issue-tracker.md`（若存在）；分拣角色见 `docs/agents/triage-labels.md`。

---

## 全局硬规则（所有类型都必须遵守）

1. **开工前**：必须明确「改什么、怎么算做完」——来自 **issue 路径** / **PRD 路径** / **用户当轮写清的验收标准**，三选一。
2. **收工前**：必须运行 **`.harness/knowledge/index.yaml`** 里目标 domain 的 **`validate_commands`**。
3. **禁止**除非用户明确要求而执行 `git commit`。
4. **未在本文件当前章节列出的步骤，默认不做**（不要自行加 landscape、product-council 等，除非用户明确要求）。

---

## 先选哪一种工作类型

按顺序判断，命中即进入对应章节：

1. 已有 **`Status: ready-for-agent`** 的 issue（或等价「可供 Agent 直接实现」的工单）？→ **[§ 领 issue 实现](#领-issue-实现)**
2. 用户明确 **修 bug / 改一两处 / 小需求**？→ **[§ 修 bug 与小改](#修-bug-与小改)**
3. 用户明确 **重构 / 整理结构 / 不改（或极少改）对外行为**？→ **[§ 旧功能重构](#旧功能重构)**
4. 其余（新能力、新模块、新功能）？→ **[§ 新功能开发](#新功能开发)**

---

## 新功能开发

**适用**：新增能力、新模块、对外行为或 API 有实质扩展。

| 阶段 | 产出 | 技能/手段 | 可跳过条件 |
|------|------|-----------|------------|
| ① 界定范围 | 一句话目标 + 明确「不做什么」 | **grill-me**（或用户已书面说明） | 用户已给出完整验收标准时可跳过拷问 |
| ② 成文 | PRD 或等价规格 | **to-prd**；存放见 issue 跟踪器（如 `.scratch/<feature>/PRD.md`） | 单 PR、范围极小、用户不要文档时 |
| ③ 拆工 | 可领取的实现 issue | **to-issues** → **triage** → `ready-for-agent` | 单步能在一个 issue 内完成且已写清时可合并 ②③ |
| ④ 实现 | 代码 + 通过验证 | 见 `CLAUDE.md`「实现阶段」 | 不可跳过 |
| ⑤ 收尾 | 知识库与结构同步 | **learn**（结构性变更时**须**做） | 仅改一行、无模式变化时可跳过 |

**禁止**：无范围就跨多目录大改；跳过 ① 直接写 PRD（范围不清时）。

---

## 旧功能重构

**适用**：改善结构、可测性、边界划分；**不**以新功能交付为主。

| 阶段 | 产出 | 技能/手段 | 可跳过条件 |
|------|------|-----------|------------|
| ① 界定重构 | 改什么、不改什么（尤其对外行为） | **grill-with-docs**（有 CONTEXT/ADR）或 **improve-codebase-architecture** | 用户已写清重构边界时 |
| ② 记录决策 | ADR 或 issue 说明「改前/改后结构」 | 更新 `docs/adr/` 或 `.scratch/.../issues/` | 纯局部提取函数、无架构含义时 |
| ③ 拆工（大重构） | 分步 issue | **to-issues** | 单 PR 可完成时 |
| ④ 实现 | 行为不变前提下的代码变更 + 验证 | 见 `CLAUDE.md`「实现阶段」；测试/类型检查必须通过 | 不可跳过 |
| ⑤ 收尾 | 更新知识库 | **learn**（**建议视为必选**） | — |

**禁止**：用 **to-prd** / **product-council** 当重构入口；未说明「行为是否允许变」就改公共 API。

---

## 修 bug 与小改

**适用**：缺陷、文案、配置、单点逻辑；范围在用户一句话内能说清。

| 阶段 | 产出 | 技能/手段 | 可跳过条件 |
|------|------|-----------|------------|
| ① 定位 | 复现路径或根因假设 | **diagnose**（难解/性能回退时**须**走满循环） | 原因已明确时 |
| ② 实现 | 修复 + 验证 | 见 `CLAUDE.md`「实现阶段」 | 不可跳过 |
| ③ 收尾 | — | **learn** | 默认跳过 |

**禁止**：扩 scope 做「顺便重构」；未复现就大面积改逻辑（除非用户接受 speculative fix）。

---

## 领 issue 实现

**适用**：工单已存在，且规格足够 Agent 开工（本地 markdown 为 `Status: ready-for-agent`）。

| 阶段 | 产出 | 技能/手段 | 可跳过条件 |
|------|------|-----------|------------|
| ① 读工单 | 确认范围与验收标准 | **Read** 指定 issue/PRD 路径 | 不可跳过 |
| ② 实现 | 代码 + 验证 | 见 `CLAUDE.md`「实现阶段」 | 不可跳过 |
| ③ 回写 | issue 评论或状态（按跟踪器约定） | 人工或 **triage** 流转；Agent 不擅自改 `Status` 除非用户要求 | 视团队约定 |
| ④ 收尾 | 知识库 | **learn** | 无结构性变更时跳过 |

**禁止**：擅自扩大 issue 范围；把 `needs-triage` / `needs-info` 的 issue 当可开工作业。

---

## 实现阶段（技术向，各类型共用）

与 `CLAUDE.md` 一致，在通过上文「开工前」门禁之后执行：

1. 读取 **`.harness/knowledge/index.yaml`** 中目标 domain 的 `rules` 与 `skills[]`
2. 先读 **`.harness/knowledge/query/<domain>.md`**（若存在）
3. 在 `domain.path` 下找相似实现
4. 按分层实现并运行 **`validate_commands`**
