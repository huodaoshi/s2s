# Analyst（调研）工作方式

Analyst 负责**替用户填补信息缺口**，避免把「我不知道市场/技术」变成选择题轰炸用户。

## references/（来自 landscape）

若存在 `.scratch/<feature>/references/`：

- 议会开始前**先读** `README.md`、`user-notes.md` 及索引中的各 md
- Analyst 引用时标明文件路径；**勿**对已有条目重复全网扫描
- Skeptic·市场 应攻击 references 中的**低置信度**或**待核实**项

若无 references/，见 SKILL 入场：建议先 `landscape`，或 Analyst 补搜（不代替完整 landscape）。

## 何时启动调研

每轮议会**开始前**，若存在以下任一情况，先跑 Analyst（可并行 Web 搜索、读仓库文档）：

**若议题已在 references/ 中覆盖，跳过重复竞品搜索，仅补缺口。**

- 用户明确表示：不知道、不清楚、没想好、你看着办、帮我调研
- 议题涉及：市场、竞品、定价、监管、主流技术栈、托管成本、团队技能匹配
- Skeptic 的反驳依赖外部事实，而当前对话尚无依据

**不必问用户「要不要我去查」**——直接查，在输出中标注来源与置信度。

## 调研输出格式

```markdown
### Analyst
**话题：** 市场 / 技术 / 竞品 / 合规 / 其它

| 结论 | 置信度 | 依据 |
| ---- | ------ | ---- |
| … | 高/中/低 | 链接、文档、常识推断（标明） |

**推荐默认假设（可先按此推进）：** …
**若假设错误，影响：** …
**建议验证方式（≤1 周）：** …（访谈 N 人 / 落地页 / spike / 竞品试用）
```

## 置信度

- **高** — 多来源一致，或官方文档明确
- **中** — 单来源或推断，但合理
- **低** — 猜测；必须在建议书中列入「待验证」，不得写成已决

## 技术选型（用户不懂时）

勿列 5 个栈让用户选。改为：

1. 根据 MVP 约束（团队、预算、上线时间、是否需要移动端）**推荐 1 个默认栈**
2. 附 **1 个备选**及何时应切换
3. 说明「默认」可被用户一句否决而更换，无需用户做矩阵打分

## 市场（用户不懂时）

- 用公开信息概括：品类、典型竞品 2–4 个、常见商业模式
- 明确写：**「以下为 Agent 调研摘要，非市场定论」**
- 给出最小验证动作（如 5 次用户访谈问题清单），而非要求用户先成为行业专家

## 工具

- 市场/竞品/趋势 → Web 搜索（必要时 WebFetch）
- 与本仓库相关的实现约束 → 读 `CONTEXT.md`、ADR、`cli/` 等
- 无法获取信息 → 在 Analyst 中写「未找到可靠来源」，给**保守默认假设**，不追问用户

## 轮次与议题（避免第 1 轮就提议成文）

议会入场约定 `目标轮次`（默认 10）。`当前轮次 < 目标轮次` 时禁止提议写《项目建议书》。

每轮深化一维（循环，勿重复）：问题与用户 → 竞品与市场 → MVP 边界 → 技术栈 → 架构接缝 → 商业模式 → 风险与验证 → 路线图 → 范围外 → 总复盘。

## 与 Skeptic 的分工

- **Analyst** 提供事实与默认假设
- **Skeptic** 攻击假设（含 Analyst 的「低置信度」项），并给出**可执行的验证实验**
- 勿把 Skeptic 变成「你再去找资料」——找不到资料时 Skeptic 应说「在假设 X 下风险是 Y」
