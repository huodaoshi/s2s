# 《项目建议书》模板

议会结束后写入 `.scratch/<feature-slug>/项目建议书.md`。

```markdown
# 项目建议书：<项目名称>

**状态：** 草案 | 待评审 | 已采纳
**议会轮次：** <N>
**最后更新：** <日期>

## 议会配置

- **目标轮次：**
- **推进席：**
- **质疑席：**

## 执行摘要

3–5 句话：为谁、解决什么、建议 MVP、**Agent 推荐**技术方向、是否建议立项。

## 假设登记册

| 假设 | 置信度 | 若错误的影响 | 验证方式 |
| ---- | ------ | ------------ | -------- |
| … | 高/中/低 | … | … |

## 调研摘要（Analyst）

> 以下为 Agent 基于公开信息的摘要，非市场定论。

- **市场/品类：**
- **竞品（2–4）：**
- **常见商业模式：**

## 问题与机会

- 用户/业务问题
- 为何现在做（证据或假设）

## 目标用户与核心场景

- 画像
- 2–3 个核心场景

## 产品形态与 MVP

- 推荐形态
- MVP 包含 / 不包含

## 技术选型

### 推荐（默认）

- **栈：** …
- **理由：**
- **适合你因为：** …（团队、时间、约束）

### 备选（何时切换）

- **栈：** …
- **何时考虑换：** …

### 待验证（≤1 周可做的 spike）

- [ ] …

## 架构草图（逻辑层）

- 模块与职责
- 外部接缝
- 刻意不做

## 阶段路线图

| 阶段 | 目标 | 退出标准 |
| ---- | ---- | -------- |
| 0 发现 | | |
| 1 MVP | | |

## 风险与未消解质疑

1. …

## 建议下一步

- [ ] prototype / to-prd / 继续 product-council
```

## 写作要求

- **默认推荐**写清楚；用户未反对的假设可标「议会默认采纳」
- 低置信度不得伪装成已决
- 「待验证」须可在一周内执行，勿笼统「需进一步调研」
